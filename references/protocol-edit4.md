# 수정4 프로토콜

파일 내용 변경(교체·삽입·삭제) 전반에 적용. 맥락 영향 범위를 5레벨(L0~L4)로 판정하고, 레벨별 게이트를 통과한 뒤 외과적 프로토콜로 실행하는 구조. str_replace 매칭 실패 → write_file fallback → 기억 기반 재구성 → 잡탕을 구조적으로 차단하기 위해 존재한다.

**3가지 모드:**

| 모드 | 정의 | 핵심 위험 |
|------|------|-----------|
| 교체 | 기존 A → 새 a로 대치 | old잔존+new공존(잡탕) |
| 삽입 | 특정 위치에 새 내용 추가 | 위치 오류, 중복 삽입, 주변 구조 훼손 |
| 삭제 | 기존 내용 제거 | 과삭제(의도 밖 내용까지 제거), 삭제 누락 |

## 레벨 판정 기준

| 레벨 | 명칭 | 맥락 영향 | 판정 기준 |
|------|------|-----------|-----------|
| L0 | QC교정 | 없음(렌더링문법만) | semantic_change=none (3조건): ❶변경유형=렌더링문법교정(빈줄추가·앵커→위키링크·콜아웃문법) ❷콘텐츠 의미변경 없음 ❸호출자 무관. 콘텐츠 변경 수반시 L1 이상 자동승격 |
| L1 | 맥락0 | 없음 | 단순 단어 치환. 주변 문맥에 영향 0 (오타, 명칭 변경 등) |
| L2 | 맥락약 | 미미 | 단어+조사 수준 치환. 문법적 정합만 확인 (용어 통일, 어미 변경 등) |
| L3 | 맥락중 | 섹션 내+관련 섹션 | 변경으로 해당 섹션 밖 다른 섹션의 텍스트가 **의미상 불일치**하게 되는 경우. 단순 용어 참조(같은 단어가 타 섹션에 등장)만으로는 L3가 아니라 L2 |
| L4 | 맥락강 | 문서 전체 | 전반적 내용 수정. 단순 갭분석으로 불충분, 내용 논의 필요 |

**판정 규칙:**
- 변경이 해당 섹션 밖으로 파급 → L3 이상
- 파일 내 3개↑ 섹션에 영향 → L4
- 애매할 때 → 상위 레벨로 승격 (escalate-up default). 에이전트가 낮은 레벨로 판정해 게이트를 우회하는 것이 가장 위험하기 때문
- 복수 파일 수정 시: 개별 레벨 max + 파일 간 의존관계 존재 시 1단계 승격

## 게이트

| 레벨 | 보고 | 진행 | 형식 |
|------|------|------|------|
| L0 | 무보고자율 | 자율 | view 반영확인 1회만. POST_VERIFY·전파맵 면제 |
| L1 | 보고+즉시진행 | 자율 | `[수정4-L1] {변경내용 1줄}` |
| L2 | 보고+즉시진행 | 자율 | `[수정4-L2] {변경내용 1줄}` |
| L3 | **보고 먼저** | **컨펌 후 진행** | Before/After 테이블(위치·변경전·변경후 3열) + 파급범위 요약 1줄. 형이 "진행/ㄱ/OK" 등 긍정 응답 시 컨펌 |
| L4 | **보고 먼저** | **논의 후 합의** | 변경 방향·근거·영향 범위를 서술. Before/After가 아니라 내용 논의. 합의 후 실행 계획 재수립 |

**배치 컨펌 (L3 다파일):**
동일 작업 맥락에서 L3 수정이 2건↑ 발생할 때, 건별 컨펌 대신 통합 Before/After 테이블로 1회 제시할 수 있다.

| 조건 | 행동 |
|------|------|
| 동일 작업 맥락 + L3 × 2건↑ | 통합 Before/After 테이블(파일·위치·변경전·변경후 4열) 1회 제시 |
| 형 응답: "전체 ㄱ" / "전부 진행" | 전 건 일괄 실행 |
| 형 응답: 건별 지정 ("2번만" / "1,3번 진행") | 지정 건만 실행, 나머지 보류 |
| L4 혼재 | L4는 배치 불가 — 별도 논의. L3만 배치 가능 |

## 실행 (외과적 프로토콜 호출)

모든 레벨·모든 모드 공통으로 외과적 트리거의 3단계를 적용:

1. **변경범위 최소:** 변경 대상만 정확히 특정. 불필요한 주변 텍스트 확장 금지
2. **주변구조 불변:** 실행 후 view로 전후 10줄 확인 — 의도치 않은 삭제/중복/구조훼손 발견 시 즉시 롤백
3. **전체정합성:** 해당 파일 내 변경 결과가 문서 전체와 충돌 없는지 grep 검증

**모드별 실행 규칙:**

| 모드 | 실행 전 | 실행 | 실행 후 |
|------|---------|------|---------|
| 교체 | view로 대상 구간 확인 → old_str 정확히 추출 | str_replace (old→new) | grep old 잔존 검색 |
| 삽입 | view로 삽입 위치 전후 확인 → 앵커 텍스트 특정 | 앵커 뒤/앞에 new 삽입 | view로 위치 정확성 + 중복삽입 확인 |
| 삭제 | view로 삭제 대상 범위 확인 → 경계 명확히 특정 | 대상만 제거 | view로 과삭제 여부 + 주변구조 무결성 확인 |

**str_replace 실패 방지 규칙 (교체·삭제 모드):**
- replace 실행 전: view로 대상 구간 확인 → raw 텍스트에서 old_str 정확히 추출 (줄번호 prefix 제외)
- 매칭 실패(no match) 시: silent하게 다른 방식으로 쓰기 금지 → 실패를 보고 → view 재확인 후 old_str 재추출하여 재시도
- write_file fallback 금지: str_replace 실패 시 write_file로 전체 재작성하면 기억 기반 재구성이 되어 잡탕의 원인이 됨. 반드시 view 결과 기반으로만 작성

**삽입 모드 추가 규칙:**
- 삽입 위치를 앵커 텍스트(삽입 직전/직후의 기존 텍스트)로 특정. 줄번호만으로 위치 지정 금지 — 줄번호는 다른 수정으로 변동될 수 있기 때문
- 삽입으로 인해 들여쓰기·섹션 경계·참조·목차가 훼손되지 않는지 view로 확인

## POST_VERIFY (전 레벨 공통)

모든 수정 후 반드시 실행. 스킵 = FAIL.

**L0 경량 POST_VERIFY (★ 최소 — view 1회만):**
L0(QC교정)은 **view 반영확인 1회**만 수행한다. ①②만 확인, ③④ 면제. grep 잔존검색도 면제. 이유: G1~G5 렌더링 교정(빈줄 추가, 앵커→위키링크 등)은 콘텐츠 의미 변경이 없으므로 검증 비용이 위험을 초과한다.

**L1 경량 POST_VERIFY (★ 이것만 하면 된다):**
L1(맥락0)은 **view 반영확인 1회**만 수행한다. 이하 전부 L2↑ 전용 — L1에서 실행하면 과잉:
- ❌ ③전파맵 조회 → L2↑
- ❌ ④참조 무결성 grep → L2↑
- ❌ ⑤결과 보고 테이블 → L2↑
- ❌ grep old 잔존 전수검색 → L2↑
L1은 view 1회로 ①중복 ②누락만 확인하고 `[수정4-L1 POST_VERIFY OK]` 1줄 보고로 종료. 이유: 오타·명칭 치환 수준은 파급이 없으며, 검증 비용이 변경 위험을 초과하면 과잉이다.

**공통 (전 모드):**
1. **중복 검사:** 동일 내용이 의도치 않게 2회↑ 존재하는지 확인
2. **누락 검사:** 의도한 변경이 정확히 반영되었는지 view로 확인
3. **전파맵 조회:** UP §F.전파맵 참조 → 현재 파일의 연쇄 갱신 대상 리스트 출력 → 미갱신 대상 존재 시 보고. 전파맵에 없는 의존관계 발견 시 전파맵에 추가 제안
4. **참조 무결성:** 변경된 파일을 참조하는 @ref·위키링크·REF 경로가 유효한지 grep 검증
5. **결과 보고:** L1·2는 자동검증+자동수정 후 1줄 보고. L3·4는 검증결과를 형에게 테이블로 보고

**모드별 추가 검증:**

| 모드 | 추가 POST_VERIFY |
|------|------------------|
| 교체 | grep old 잔존 전수검색 → 1건↑ = FAIL. **잡탕 판정:** old잔존+new공존 = 최악의 실패, 부분 성공 없음 |
| 삽입 | 삽입 위치 정확성 확인(의도한 앵커 직후/직전인지). 주변 구조(들여쓰기·섹션 경계·참조) 무결성 확인 |
| 삭제 | 과삭제 확인(의도 밖 내용까지 제거되지 않았는지). grep으로 삭제 대상이 완전히 제거되었는지 확인 |

## CASCADE 규칙

### CASCADE_CAP (연쇄 수정 재컨펌 방지)
1차 수정 confirm 후 cascade로 발생하는 2차 수정은 레벨 불문 보고+진행(재컨펌 불요). 3차↑ = 1줄 보고만.
이유: 연쇄 수정마다 컨펌을 받으면 작업이 끊기고, 형의 의도는 이미 1차에서 확인됨.

### CASCADE_CACHE (전파맵 캐시)
동일 세션 내 연속 수정시, 직전 전파맵 조회 후 전파맵 자체가 미변경이면 → 2차↑ 전파맵 조회 스킵.
캐시 무효화 조건: 스킬·UP 수정, 전파맵 MAP 등록파일 수정시.
이유: 같은 세션에서 같은 파일을 반복 수정할 때 매번 전파맵을 조회하면 과잉.

### SECTION_DEF (섹션 단위 정의)
- heading(h1~h4) 또는 callout 단위 = 1섹션
- frontmatter·YAML = 별도 섹션
- heading 없는 문서 → 빈행 구분 블록 = 1섹션
이유: L3↑ 판정에서 "섹션 밖 파급"의 기준이 되므로, 섹션의 경계가 명확해야 한다.

## 금지 행동

| # | 금지 | 해당 모드 | 이유 |
|---|------|-----------|------|
| 1 | 중복수정 | 교체 | 이미 바뀐 곳을 또 바꿈 → 이중 적용 |
| 2 | 기존잔재남기기 | 교체 | old가 살아남은 채 new 추가 → 잡탕 |
| 3 | 수정누락 | 전체 | 바꿔야/넣어야/지워야 할 곳을 안 함 → 불완전 |
| 4 | 착오수정 | 전체 | 대상이 아닌 곳을 변경 → 오염 |
| 5 | 조용히끼워넣기 | 전체 | 형이 요청하지 않은 내용을 슬쩍 추가 → 신뢰 훼손 |
| 6 | 위치 오삽입 | 삽입 | 의도한 위치가 아닌 곳에 삽입 → 구조 혼란 |
| 7 | 과삭제 | 삭제 | 의도 밖 내용까지 함께 제거 → 데이터 손실 |
| 8 | 중복삽입 | 삽입 | 동일 내용 2회↑ 삽입 → 중복 |
